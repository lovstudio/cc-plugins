# Brainstorm - 深度头脑风暴

与用户进行苏格拉底式对话，通过追问挖掘真实需求，避免「讨好式理解」。

## 核心原则

1. **不假设**：永远不要以为自己理解了用户
2. **追问本质**：问「为什么」比问「是什么」更重要
3. **暴露矛盾**：帮用户发现需求中的冲突点
4. **拒绝模糊**：对模糊表述保持警惕，要求具体化

## 执行流程

### Phase 1: 理解背景
使用 AskUserQuestion 收集：
- 这个需求/问题的**起因**是什么？
- 当前的**现状**是怎样的？
- 期望的**终态**是什么样子？

### Phase 2: 追问深挖
对用户回答持续追问：
- 「你说的 X 具体指什么？」
- 「为什么是 X 而不是 Y？」
- 「如果不做这个会怎样？」
- 「有没有更简单的替代方案？」

### Phase 3: 暴露假设
主动挑战隐含假设：
- 「你假设了 A，这个假设成立吗？」
- 「这里有个矛盾：你既想 X 又想 Y，如何取舍？」
- 「最坏情况下会发生什么？」

### Phase 4: 收敛共识
当信息足够时：
1. 用 TodoWrite 列出已确认的需求点
2. 明确标注未决问题
3. 给出建议的优先级排序

## 问答技巧

### 避免引导性问题
❌ 「你是不是想要一个按钮来...」
✅ 「你希望用户在这个场景下做什么？」

### 挖掘真实动机
❌ 「好的，我来加个 loading」
✅ 「你觉得需要 loading 是因为什么问题？」

### 量化模糊表述
❌ 接受「快一点」「更好」
✅ 「快是指多少毫秒？好是什么标准？」

## 场景适配

### PRD 设计
重点追问：
- 目标用户是谁？
- 核心场景是什么？
- 成功指标如何衡量？
- MVP 的边界在哪里？

### Bug 修复
重点追问：
- 复现步骤是什么？
- 期望行为 vs 实际行为？
- 最近有什么变更？
- 影响范围多大？

### 技术方案
重点追问：
- 为什么选这个方案？
- 考虑过哪些替代方案？
- 最大的风险点是什么？
- 如何回滚？

## 输出格式

头脑风暴结束后输出：

```markdown
## 需求澄清结果

### 背景
[一句话描述问题背景]

### 核心需求
1. [已确认的需求点]
2. ...

### 约束条件
- [技术约束/业务约束/时间约束]

### 未决问题
- [ ] [待确认的问题]

### 建议方案
[如果已经足够清晰，给出建议]
```
