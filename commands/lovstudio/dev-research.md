# Dev Research - 技术选型调研与研发计划

## 输入
用户描述一个软件研发需求（自然语言）。

**参数格式**: `<需求描述>`

## 五种复用策略

在调研时，对每个模块/功能按以下策略分类：

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **A. 参考自研** | 学习思路，代码从零写 | 核心差异化模块、安全要求高 |
| **B. Fork修改** | Fork开源项目，改核心逻辑 | 架构匹配度>70%、只改局部 |
| **C. 库集成** | pip/npm install，用公开API | 通用基础设施、成熟稳定库 |
| **D. 混合提取** | 从多个项目提取最佳零件组装 | 需要特定功能但不需整个框架 |
| **E. 协议复用** | 复用协议/接口标准，代码自主 | 需对接外部生态（MCP/OpenAI等） |

### 决策树

```
这是核心差异化能力？
├── 是 → 策略 A
└── 否 → 有成熟库可直接用？
         ├── 是 → API匹配度>80%？
         │        ├── 是 → 策略 C
         │        └── 否 → 策略 D
         └── 否 → 有架构相似的开源项目(>70%)？
                  ├── 是 → 只需改局部？
                  │        ├── 是 → 策略 B
                  │        └── 否 → 策略 A+D
                  └── 否 → 需对接外部生态/协议？
                           ├── 是 → 策略 E
                           └── 否 → 策略 A
```

## 流程

### Phase 1: 需求澄清

用 AskUserQuestion 澄清关键问题（最多2轮）：

**必问**（未在参数中提供时）：
- 目标用户和核心场景
- 技术栈偏好/限制（语言、框架、部署环境）
- 预算约束（纯开源 vs 可付费API vs 无限制）
- 时间预期（MVP多久交付）

**可选**（复杂需求时）：
- 性能/并发要求
- 数据隐私合规要求
- 是否需要与现有系统集成

### Phase 2: 模块拆分与调研

1. **拆分功能模块**：将需求拆为独立模块（如：认证、支付、存储、通知...）

2. **逐模块走决策树**：对每个模块套用决策树，确定复用策略

3. **搜索现有方案**：
   - WebSearch 搜索 `"<关键词> open source"` / `"<关键词> API"` / `"<关键词> SaaS"`
   - 评估维度：GitHub stars、最近更新、定价、技术契合度

4. **KISS 原则兜底**：能用现成方案解决的，别自己写

### Phase 3: 输出研发计划

输出到 `./dev-research-<项目名>.md`：

```markdown

<一段话总结>
```

### Phase 4: 确认

展示计划摘要，询问是否调整。

## 注意事项
- 给出**具体项目/产品名和链接**，不泛泛而谈
- 每个推荐方案说明**为什么选它而不是其他**
- 成本估算基于**实际定价页面**
- 排除方案必须记录原因，避免重复调研
- 如果需求可以用一个现成工具解决，直接说
